# Switch grid editor UX redesign (stem-driven cyclers)

Rework the **authoring editor** for the existing `SwitchGridElement` so creators are not confused. The data model, student render, check endpoint, and transfer are all shipped and **unchanged** — this touches only the edit-form partial, the editor JS, the CSS, and the tests that cover them.

## Purpose

The shipped editor (`_edit_switchgrid.html` + `switchgrid_editor.js`) presents cyclers as a padded, add-only list only loosely tied to the `{{choice}}` markers in each line's stem. Observed problems (from a creator using it):

1. Every line auto-shows **one extra blank cycler** (padding) — the creator sees phantom "Option 1/2/3" blocks with no explanation.
2. An extra blank **line** is auto-appended at the bottom, with its own blank cycler — looks unintentionally created, and there is **no way to remove it**.
3. Each cycler shows **3 blank option inputs** even though the minimum is 2 — confusing for a 2-option cycler.
4. **No remove affordance** anywhere: the only way to "delete" a row is to leave it blank and rely on silent drop-on-save (undiscoverable).
5. **Add option / Add cycler buttons overlap** (CSS/layout bug).

Root cause: the editor was built by mechanically mirroring the switchgate editor's "padded rows + blank-dropping" pattern with **no design pass** over the authoring interaction.

Goal: an editor where the relationship *marker → cycler* is obvious, nothing appears that the creator didn't ask for, everything shown can be removed, and the layout reads as a clear nested hierarchy.

## Design overview

### Cyclers are stem-driven (the core change)

The stem is the single source of truth. The editor counts the `{{choice}}` markers in a line's stem and renders **exactly that many** cycler blocks — no more, no fewer. There is **no "Add cycler" button** and no padding. This eliminates the **missing-block** mismatch class (you can never have fewer/more blocks than markers on screen). One residual case remains and is handled explicitly: a creator can leave a cycler's options **all blank**, which the server `clean()` drops → a count mismatch. The JS **submit guard** (see Options / the submit flow) catches that first with a clear per-cycler message, so the confusing server "marker≠cycler count" error is never what the creator sees.

- A stem with **zero** markers → a **static line** with no cycler blocks. The redesign makes this a first-class, easily-reachable state (delete the seed's sole `{{choice}}` and a stem-only line remains — previously masked by the padded blank cycler). **Invariant to preserve/verify:** the unchanged `clean()` **keeps** a stem-only, zero-cycler line (it is kept because its stem is non-blank, contributing `cyclers: []`), so long as the grid has ≥ 1 cycler somewhere (the existing ≥1-cycler-total rule). It is dropped only when the stem is *also* blank. This must hold on the save/re-load round-trip, not just visually — see Testing.
- The JS re-derives blocks from the stem on `input` (debounced ~150 ms) so blocks appear/disappear as the creator types or deletes markers.

**Cycler indexing (positional, dense, tail-managed) — the single coherent rule:**
- A line's cycler blocks are always **dense and positional**: exactly N blocks for N markers, indexed `c0 … c(N-1)` in DOM order, so the visual "Cycler 1/2/…" label = block position = the marker's left-to-right position. There are no index gaps at the cycler level (this differs from the shipped append-only `j`; the redesign makes `j` positional). The unchanged server `clean()` tolerates this — it sorts/compacts cycler indices, and dense 0..N-1 is the already-compacted case.
- **Reconcile matches the block COUNT by adding/removing at the tail only.** When the marker count rises, blank blocks are appended (`c(old) … c(new-1)`); when it falls, trailing blocks are removed. Blocks `c0 … c(min-1)` are **never renamed and never touched** — their field names, option values, and checked radio stay exactly as they are. (So no live re-indexing of existing blocks is ever needed — a deliberate simplification over "true positional with shift".)
- **Consequence — end edits are lossless, middle edits are index-aligned (accepted, symmetric edge):** adding or deleting a marker at the **end** of the line preserves every block's data intuitively. Inserting or deleting a marker in the **middle** keeps each block's data at its position (tail block appended/removed), so the marker↔block *alignment* shifts by one while the data stays put. This is the accepted edge case for **both** insertion and deletion; it is rare (lines are authored left-to-right), always visible (the creator sees the blocks), and correctable in place. The spec does not attempt smart middle-alignment.

**Preserving work across marker deletes (no accidental data loss):**
- A removed (tail) block's entered options + answer are **stashed** so a transient delete-then-retype is non-destructive within the session. The stash is a JS-only, per-editor-instance map keyed by **`(line index, cycler block index)`** — line index is required because block indices repeat across lines, and per-editor scoping is required because the editor DOM can be replaced by an AJAX partial (validation-error re-render / edit load). On a fresh `[data-switchgrid-editor]` root the stash starts empty; removing a whole line **purges that line's stash entries** (and line indices are append-only / never reused, so a purged entry cannot resurface). When reconcile re-adds a tail block whose `(line, block)` key is stashed, it repopulates that block.
- Labels ("Cycler 1", "Cycler 2", …) are positional UI **rendered by the JS and recomputed on every reconcile** by visible block position (not a server-rendered constant).

### Options within a cycler

- A cycler block starts with **2** option inputs (the enforced minimum, not 3).
- Each option row: a **correct-answer radio** (one radio group per cycler) + the option text input + a **remove (×)** button.
- An **"Add option"** button appends a row.
- Removing is blocked below **2** options (the × is disabled/hidden when only 2 remain), so a cycler can never be authored invalid.
- No auto-padded blank option row.
- **Positional radio-value invariant (load-bearing — the server reads the answer by position):** the server `clean()` collects a cycler's options via `getlist("line-{i}-c{j}-opt")` (all option inputs share one name, positional order) and reads `line-{i}-c{j}-ans` as the **0-based position** of the correct option (then remaps only for dropped blanks). Therefore, after **every** structural change to a cycler's option rows — add option, **remove option**, and cycler-block creation — the JS MUST **re-sequence every radio's `value` to its current DOM position (0…n-1)**. (The shipped JS already does this on add-option; the redesign extends it to remove and creation.)
- **A freshly-seeded/JS-created cycler starts with NO radio checked (M2):** the create seed, an Add-line cycler, and a new-marker block all begin unselected (matching the server's `answer = -1` render), relying on the server backstop to require a selection on save. No block auto-pre-checks option 0 — this keeps all creation paths consistent.
- **Removing the currently-checked option:** the cycler is left with **no radio selected**; the JS does not auto-select a replacement. On save the unchanged server backstop ("select the correct option in every cycler") flags it, so the creator must pick a new correct option — a visible, correct outcome rather than a silent wrong answer. Radio values still re-sequence per the invariant above.

### Lines

- An **"Add line"** button appends a new line (seeded like the first — see create default).
- Each line is a **card** with a **remove (×)** control.
- **No auto-appended blank line.** Minimum one line (the × is disabled/hidden when only one line remains).
- **Line-index derivation (load-bearing — must NOT use DOM position).** Unlike cyclers (kept dense and tail-managed), lines become genuinely **gappy** after a middle-line ×-removal (e.g. `line-0`, `line-1`, `line-2` → `line-0`, `line-2`). A new line's `i` MUST come from a **per-editor-instance monotonic counter (or `max(existing line indices)+1`)**, never from DOM child count/position — the naive "index = child count" would pick `2` after that removal and collide with the surviving `line-2-*`, and the server's set-dedup `_line_indices()` would silently merge the two lines' fields into one corrupted line on save (no error surfaced). Post-removal DOM line indices are **intentionally gappy**; the server compacts them (sorted-set discovery + positional `line_rows` re-render), exactly as it already does for cyclers. This is the same positional-name-uniqueness invariant pinned for cyclers/radios, applied to lines.

### Create default (new empty element)

A brand-new Switch grid opens with **one line, its stem pre-seeded with a short worked example containing one `{{choice}}` marker**, and therefore **one cycler block with two empty option inputs**. The creator immediately sees the pattern (marker → cycler → options) without reading docs, and edits the example in place. The seed stem is the fixed literal `2 {{choice}} 2 = 4` (see the seed constant in Architecture); the option inputs are empty. "Add line" seeds new lines the same way (one marker + one 2-option cycler) so the pattern stays visible.

The seeded default is a **template to fill in, not directly savable**: its two option inputs are empty, so `clean()` drops the empty cycler and returns the "add at least one cycler with options" / marker-mismatch errors if the creator saves it untouched. That validation-error path on an untouched create is **expected behavior**, not a regression — and the 422 re-render (below) preserves whatever the creator did type.

### Visual / layout (executed with the frontend-design skill)

- A line renders as a **card**: the stem textarea on top, then its cycler blocks **indented** beneath it, each option row indented within its block — so the line → cycler → option hierarchy is visually unambiguous.
- Add/Remove controls are **grouped and labeled per level** and laid out so nothing overlaps (fixes the button-collision bug). Remove (×) controls are small, right-aligned, and clearly secondary.
- Consistent with the app's token-driven design system in **both light and dark themes** (reuse existing `--` tokens; no invented colors). Follow the frontend-design skill for typography, spacing, and hierarchy so it does not read as a templated default.
- Verified by screenshots (light + dark) with a self-critique pass before shipping, per the project's "every view ships styled" / screenshot-verification practice.

## Architecture / components

Files touched (all under `courses/`, `templates/`, `core/`):

- **`templates/courses/manage/editor/_edit_switchgrid.html`** — rewritten. Emits: the single **Instruction (`prompt`) field** with its `<label>` (unchanged form field), a `[data-switchgrid-editor]` root, a `[data-lines]` container of line cards, and **three `<template>` nodes the JS clones** — a **line template**, a **cycler-block template** (its `[data-options]` wrapper, the "Cycler N" label region, the "Add option" button, and no starter option rows — the JS seeds two option rows into a freshly-cloned block by cloning the option-row template twice), and an **option-row template** (radio + text input + × ). The `__i__`/`__j__` placeholders are rewritten to the target line/cycler indices on clone (as today) in **all three** templates — including the **line template** (its `line-__i__-stem` and its cyclers' names get `__i__` → the new monotonic line index from the line-index rule). Server-rendered initial state comes from `SwitchGridElementForm.line_rows()` but **without padding** — the partial renders only the rows that exist. Field names stay exactly `line-{i}-stem` / `line-{i}-c{j}-opt` / `line-{i}-c{j}-ans`; the form parse and its gap/blank compaction are unchanged.
- **Seed constant (single source):** a Python constant `_SG_SEED_STEM = "2 {{choice}} 2 = 4"` (a fixed math literal, **not** translated) is the one source of the create/Add-line seed. `line_rows()` uses it for the create line's stem; the **line `<template>`'s** stem textarea is server-rendered with the same constant (passed via template context, not hardcoded in two places) so the JS-cloned "Add line" and the Python create path always seed identically. **Correctness note (not just DRY):** the seed must be injected via **context** (or wrapped in `{% verbatim %}` / `{% templatetag %}`) — a literal `2 {{choice}} 2 = 4` written directly in the Django template would parse `{{choice}}` as a template variable and render `2  2 = 4`, silently breaking the seed (a template-parsing gotcha this project has hit before). Do not "optimize" the context-pass back into an inline literal.
- **`courses/static/courses/js/switchgrid_editor.js`** — reworked from "add-line/add-cycler/add-option button cloning" to: (a) **stem-driven cycler derivation** (count markers on debounced input, add/remove cycler blocks at the tail, with the per-editor stash), (b) **remove (×)** handlers for options and lines, (c) **Add option** and **Add line** handlers, (d) min-guards (≥2 options, ≥1 line), (e) radio-value re-sequencing after every option add/remove and on block creation (per the positional-value invariant). Still a single document-level delegated listener plus a delegated stem-`input` listener (robust to AJAX-injected partials).
  - **Reconciliation must also fire without an `input` event (I6):** "Add line" (and initial hydration of any JS-injected line) does not emit `input`, so the handler explicitly **runs reconcile on the new line** (calls the reconcile function directly, or dispatches a synthetic `input`) so its seeded `{{choice}}` materializes the promised cycler block. On page/partial load, all present lines are reconciled once to render their cyclers.
  - **Flush pending reconcile on submit (I2):** if the creator types a new `{{choice}}` and clicks Save within the ~150 ms debounce window, the POST would carry N markers but N-1 cycler blocks and the server would reject on marker≠cycler count — a legitimately-authored grid wrongly refused. So the editor **flushes any pending debounced reconcile (or runs reconcile synchronously across all lines) on form submit**, before the POST, so the DOM matches the stems. This eliminates the *missing-block* mismatch.
  - **All-blank-cycler submit guard (prevents the confusing server mismatch):** after the flush, the submit handler checks each cycler block for an **all-blank options** state (every `line-{i}-c{j}-opt` empty). Because the server would silently *drop* such a block and then reject the whole line with a "marker≠cycler count" error — which is nonsense to a creator who sees the block on screen — the JS instead **blocks submit and shows a clear inline per-cycler message** (e.g. "Cycler N: fill in its options, or remove its {{choice}} marker"), scrolling to / focusing the offending block. This turns the one residual mismatch case into an actionable message instead of the confusing server error. (The server `clean()` checks remain the authoritative backstop for non-JS / direct POSTs.)
  - **JS marker count == server marker count (I10):** the helper counts the literal `{{choice}}` on the **raw `rte-source` textarea value** — the *same* string the server receives and parses (`parse_stem_multi(strip_sentinel(sanitize_html(raw)))`). This holds because `{{choice}}` is plain text that `sanitize_html`/`strip_sentinel` and the RTE leave untouched (no splitting/escaping/wrapping of the literal). State this assumption explicitly; if the RTE ever transformed the literal, client and server counts could diverge and the "mismatch is unauthorable" guarantee would break.
- **`core/static/core/css/app.css`** — the `.el-editor--switchgrid` block reworked for the nested-card layout, indentation, and non-overlapping grouped buttons; light + dark.
- **Form (`courses/element_forms.py` `SwitchGridElementForm`)** — `clean()` and the `line-{i}-c{j}-*` parsing are **unchanged** (they already compact blanks/gaps and remap the answer, which the new JS relies on). `line_rows()` is adjusted so padding minimums no longer inflate the editor: on an unbound (create) form it returns the single seeded line (one marker → one cycler → **exactly two** empty option inputs); on a bound/edit form it returns **exactly the posted/stored lines, cyclers, and options** — no `+1` blank line, no `+1` blank cycler, no padded option rows. So an edited cycler with 4 stored options renders 4 rows; only the empty seed case starts at 2. The `_SG_MIN_*` padding constants are retired (the ≥2-options / ≥1-line minimums are still enforced in `clean()` and via the JS guards, just not by padding). This is the only Python change; it is behavior-preserving for the *parse* (still tolerant), changing only what the editor initially shows.

No changes to: the model, migration, `render_switch_grid` (student view), `switchgrid_check`, transfer, or `switchgrid.js` (the student enhancer).

## Data flow

1. **Create:** `manage_element_add?type=switchgrid` → `SwitchGridElementForm()` unbound → `line_rows()` returns one seeded line (stem = short example with one marker; one cycler; two empty options). The partial renders it; `switchgrid_editor.js` initializes, sees one marker → one cycler block already present.
2. **Author edits stem:** on debounced `input`, the JS counts `{{choice}}` in that line's stem and reconciles the visible cycler blocks (grow/shrink from end; restore from stash on grow). Answer radios and option inputs carry the correct `line-{i}-c{j}-*` names (indices assigned append-only; the form compacts any gaps on save).
3. **Author adds/removes options or lines** via the ×/Add controls; min-guards keep it valid.
4. **Save (POST):** the form parses the indexed fields exactly as today — drops blank options, drops wholly-blank lines, compacts gaps, remaps answers, validates marker==cycler count (which the stem-driven UI now guarantees) — and persists `lines`.
5. **Validation-error re-render (422):** `line_rows()` mirrors the POSTed data (already implemented) so the creator's grid survives; the partial re-renders it, JS re-derives cyclers from the posted stems.

## Error handling

- **≥2 options / ≥1 line** enforced in the JS (disable/hide × at the minimum) *and* still enforced server-side in `clean()` (unchanged) as the backstop — the UI makes the invalid states unreachable, the server stays authoritative.
- **Marker/cycler mismatch** — the *missing-block* form is unreachable via the UI (cyclers are derived from markers); the one residual form (a materialized cycler left with all-blank options, which the server drops) is caught by the JS submit guard with a clear per-cycler message before the POST. The server `clean()` mismatch check stays as the authoritative backstop for non-JS / direct POSTs / imports.
- **Debounce / rapid typing:** block reconciliation is idempotent and keyed by index; a partial marker (e.g. half-typed `{{choi`) counts as zero until the full literal `{{choice}}` is present — no crash, blocks settle once typing stops.
- **AJAX-injected partial:** the JS uses document-level delegation (click + input), so a partial injected after page load works with no re-init (as today).
- **No-JS fallback (with its real limitation):** with JS off, the server-rendered `line_rows()` state is a submittable plain form (stem textareas + option inputs + radios), but because the "Add cycler" button is gone, a no-JS author can only submit the cycler blocks the server initially rendered — i.e. **single-marker lines** (the create default). Typing a second `{{choice}}` with JS off would add no block and the server would reject on marker≠cycler count. This is an accepted limitation: authoring in this app already assumes JS; the stem-driven reconciliation is a progressive enhancement over an intentionally-minimal no-JS floor, not full parity.
